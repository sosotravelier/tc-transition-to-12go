Summary
The meeting, aimed at deciding on a B2B implementation approach for service sunsetting in TC environments, involved discussions among participants regarding architectural constraints to maintain existing B2B clients, the deprecation plan for current infrastructure (including the removal of the SI framework), and the necessary configuration of the One to Go system to ensure data correlation and B2B-specific event collection. While considering the monolith versus microservice (adapter layer) implementation approach, which Soso Tughushi's detailed research consistently favored the microservice, the group did not reach a final decision but agreed to conduct a Proof of Concept (PC) for one API and further review Soso Tughushi's research to make an informed choice, with Oleksandr Monov leaving early due to another engagement.

Details
Meeting Purpose and Constraints: The meeting was convened to make two significant decisions prior to addressing smaller implementation details regarding the sunsetting of services in TC (Trip Connection) environments like Denali, Fuji, and others. A critical constraint for the process is ensuring that existing B2B clients are not broken, meaning all existing endpoints should remain as they are, though changes to the URL or API key are debatable (00:02:28). The contract for clients must remain the same to facilitate a painless transition (00:04:03).
Current Infrastructure and Deprecation Plan: The current architecture includes net services, a large infrastructure on AWS with multiple instances of Etna and Etna Sihost, and B2B clients calling three services: Etna for search, Denali for booking funnels (checkout, creation, confirmation, details, PDFs), and Fuji for master data (stations, POIs, operators) (00:04:03). The proposed plan involves eliminating the Trip Leg cache on the TC side, removing the SI framework due to moving to a single integration (One to Go), and sunsetting the pricing model, as price information will be available directly in the One to Go response (00:07:02).
Discussion on Kafka Events and Data Correlation: A concern was raised that merely keeping client-facing events on the Etna or Denali side after deprecating the SI host would result in a loss of correlation between supply integration side and client-facing events. The ability to correlate these events is necessary, and if the SI framework is deprecated, the One to Go system must be configured to send the necessary events so that data can be put into the same source, such as BigQuery (00:08:37).
Event Collection and B2B Client Specificity: The discussion about data events led to the conclusion that an overview of existing important events is needed to check for parallel events in One to Go. They agreed to define requirements from the data side and identify what is needed to replace current event functionality (00:11:56). It was noted that many events are sent from the BFF (Next or similar) rather than the backend, necessitating a check on how B2B-specific events are currently sent and potentially moving them from the BFF to the back end if necessary (00:12:43).
Client-Facing Notifications (Webhooks): The definition of "client-facing" events was clarified as notifications for B2B clients, specifically webhooks sent as HTTP requests when a booking changes status (e.g., cancellation). It was noted that only one of the current four types of events—booking notification—will remain relevant after the migration, and a transformer will be needed to convert data from One to Go into the necessary TC Connect language (00:13:48).
Core Service Usage and Long-Term Strategy: Even in the target design, they plan to continue using the One to Go F3 module, meaning events produced from this module and collected by ClickHouse will be preserved. It was emphasized that this project should not be considered a short-term solution; the goal is to build one cohesive system where the existing One to Go core services are replaced with the system being built, differentiating between core services and those specific to B2C (00:16:12).
Decision on Implementation Approach (Monolith vs. Microservice): Two options were discussed: adding the B2B module directly into the F3 code base (monolith) or creating a separate microservice acting as an adapter layer (00:19:04). Adding to F3 reduces HTTP latency but couples the teams with the existing PHP code base, while a separate microservice allows for language flexibility and acts as a transparent adapter layer (00:22:00) (00:27:47). The potential plan to break down the F3 monolith into microservices was raised as a factor favoring a decoupled approach (00:22:59).
The Anti-Corruption/Adapter Layer: The proposed microservice approach is viewed as an adapter layer or anti-corruption layer to protect existing clients. It was acknowledged that this layer might require some internal state for certain scenarios, such as associating client IDs with webhook URLs for booking updates, which currently exists in the Denali booking service (00:27:47).
Coupling, Development, and F3 Break Down: A major concern regarding a separate microservice is that any new feature development for search or booking would still require updates to both the F3 code and the new transformer microservice, leading to constant dual updates (00:30:46). While a plan to break down F3 exists, it is currently in the very beginning stages of planning and is not expected immediately, suggesting the microservice solution might endure for a year or two (00:33:41) (00:39:42). They also discussed whether to use a microservice merely for JSON transformation or to commit to writing in a non-PHP language, which is seen as the primary motivation for the microservice option (00:35:38).
Proposed Next Steps and PC: The group did not reach a final decision on the implementation approach and agreed that more research is required. A follow-up discussion is suggested to deep dive into specific details, including ID manipulation/encryption and features currently available in TC to determine if they are lost or need to be adapted in One to Go (00:55:16) (00:58:09). They proposed implementing a Proof of Concept (PC) for one API (like the get itinerary API) to gauge complexity and decide whether to implement it inside F3 or as a separate microservice (00:40:40).
Monolith vs. Microservice Architecture: The group discussed the difficulty in deciding between a B2B monolith and a microservice architecture, noting that the arguments presented so far were not deemed sufficient to justify a decision (00:59:01). Soso Tughushi shared that they attempted to make the decision obvious by creating various evaluation criteria, including implementation effort and infrastructure fit, but the results consistently favored the microservice option, even when coefficients were adjusted to try and justify the monolith (00:59:01).
Next Steps and Research Review: The participants agreed that more time was needed to thoroughly consider the important architectural decision, as the meeting had run out of time. Someone in Bangkok (Main HQ, 4) emphasized that Soso Tughushi's research was amazing and well-done, but the decision needed to be examined from all angles and required a detailed review of all the breakdowns and four revisions completed by Soso Tughushi (01:00:40). They intend to collaborate to find the right solution and ensure Soso Tughushi agrees with the final decision (01:01:45).
Oleksandr Monov's Departure: Oleksandr Monov had to leave the meeting early due to another engagement and asked to be contacted if the group required anything further from them. Someone in Bangkok (Main HQ, 4) and Soso Tughushi thanked Oleksandr Monov for their participation (00:59:01).
